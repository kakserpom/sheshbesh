# CLAUDE.md

Этот файл служит руководством для Claude Code (claude.ai/code) при работе с кодом в этом репозитории.

## Состояние проекта

Рабочая реализация самобытного варианта нард **Шеш-Беш (Shesh-Besh)** на Rust
(edition 2024). Все правила из спецификации ниже реализованы; есть эвристический ИИ,
TUI (интерактивный ratatui + потоковая ANSI-анимация) и **GUI на Leptos+Tauri**.

**Структура — cargo-workspace:**
- Корневой пакет `sheshbesh` — движок + TUI. TUI-зависимости (`crossterm`/`ratatui`)
  спрятаны за фичу `tui` (включена по умолчанию). Без неё крейт — **чистый движок**,
  компилируется в `wasm32-unknown-unknown` (база для GUI). Бинарник требует `tui`.
- `frontend/` (`sheshbesh-gui`) — Leptos CSR-фронтенд (WASM). Зависит от движка с
  `default-features = false`; вся игровая логика (движок + ИИ) работает в браузере.
  Перед партией — **экран настроек** (сигнал `started`): число игроков **2/3/4**
  (`active_for`) и тип каждой стороны — человек/компьютер (`humans: Vec<Side>`,
  `toggle_human`). При **четырёх** игроках выбирается режим (`teams`): **командный
  2×2** (союзники — противоположные стороны, `team_of = index % 2`: A+C против B+D;
  союзники **не едят** фишки друг друга — режим прокидывается в движок `GameState.teams`)
  или **каждый сам за себя** (FFA). Условие конца партии — `game_over`: в командном
  режиме победа, когда **обе** стороны команды завели фишки в Дом; в FFA партия идёт,
  пока не финишируют **n−1** сторон (последний — проигравший). Порядок финиша
  копится в сигнале `finished`; итог (победившая команда либо места) — `result_msg`.
  С экрана настроек доступны **«📖 Правила»** (сигнал `rules`) — структурированный
  справочник по самобытным правилам, и **«🎓 Обучение»** (`tutorial`) — пошаговый
  туториал: одна фишка проходит весь путь (ввод→Тюрьма→Луна→съедание→Дом). Каждый
  шаг — **граница реального хода** (`Lesson`: `before`-позиция + заранее заданный
  `roll`); сам ход вычисляет **движок** (`chosen_turn`→`legal_turns`), поэтому он
  гарантированно легален и соблюдает правило максимального хода (две кости играются
  как положено). Бросок происходит **до** хода (`roll_only_frames` на входе в шаг),
  затем ход анимируется через общий `commit_frames`. Шаги, чья `before`-расстановка
  задана **вручную** (скачком, а не получена ходом — `Lesson.teleport`, напр. шаг
  выкупа), помечают кадр перехода `Frame.fade`: доска **гасится и проявляется**
  (`fading` + CSS `.board-wrap.faded`, длительность `FADE_MS`), а смена позиций
  происходит «вслепую» — фишки не «перелетают» через всю доску. Перейти к шагу можно
  **самому**, сделав ход кликами (фишка → клетка, хит-зоны `target_cell`/`tut_sel`), либо ▶/◀.
  Кости и доска — те же, что в партии (сигналы `game`/`roll`).
  С экрана настроек доступен **режим разработчика** (сигнал `dev`, кнопка «🛠 Режим
  разработчика»): отдельный экран вне партии (авто-ход/авто-бросок и подсчёт финишей
  отключены) с панелью сценариев-демо (`Demo`/`DEMOS`/`demo_state`/`demo_game`) —
  предельные расстановки для проверки отрисовки (16 фишек на одном поле Луны, на одной
  клетке, в одной Тюрьме, в плену; стопки одного цвета НА ВЫХОДЕ из Тюрьмы со счётчиком
  (освобождённые «зашёл-вышел» стоят на клетке Тюрьмы, как на углу) — `Demo::PrisonStack`;
  заполненные Дома) и кнопкой **демо-анимации**
  (`demo_capture` + `anim_demo`), проигрывающей пошаговый ход со съеданием через тот же
  конвейер кадров (`commit_frames`/`play`), что и партия. Доска на этом экране —
  read-only копия игровой (без интерактивного слоя). Отдельная кнопка **«🏠 Тест: вход
  в Дом»** (`home_test` + `home_entry_test`) стартует **интерактивную** партию из
  расстановки, где фишки A стоят вплотную к Дому, с заранее заданным броском (минуя
  авто-бросок: `roll`/`turns`/`remaining` ставятся вручную) — чтобы быстро проверить
  клик «вход в Дом» по клетке-воротам.
  **Компьютер ходит обученной TD(λ)-моделью** (не эвристикой): веса встроены в бинарь
  (`include_bytes!("model.bin")`, `ai_model_for` → `Ai`; число скрытых нейронов
  выводится из длины весов в `MlpValue::from_floats`), ход выбирается
  `best_turn`/`best_forced` (жадно по ценности афтерстейта), а для варианта
  **MLP+поиск** — `best_turn_2ply` (2-плай expectiminimax, см. `src/search.rs`).
  **Алгоритм каждого компьютера выбирается на экране настроек** — сигнал
  `algos: [Algo; 4]` (по индексу `Side`, по умолчанию MLP); для каждой
  стороны-компьютера показывается селектор **MLP / MLP+поиск / Linear / Эвристика**
  (`Algo::ALL`; `Algo::is_search` → ветка с `best_turn_2ply`). Агент строится под
  ходящую сторону: `enum Ai { Mlp(MlpValue), Linear(LinearValue), Heuristic }`
  реализует `Value` (делегирование), `ai_model_for` принимает выбранный `Algo`
  (`Search` строит ту же MLP-оценку, поиск надстраивается на вызове). Линейные веса — отдельные файлы
  `frontend/src/model_lin{,_3p,_4p,_4p_teams}.bin` (`examples/export_linear.rs`),
  MLP-веса — `model{,_3p,_4p,_4p_teams}.bin`. Модель — **MLP** (`MlpValue`,
  **h=48**) над обогащёнными признаками (см. `encode.rs`: `mobile[6]` — играбельность
  каждой кости 1..6, `corner` — фишки на углах, `blots`/`contact` — уязвимые одиночки
  и блоты под прямым боём). Веса генерирует
  `examples/export_model.rs` (`cargo run --release --example export_model`) — обучение
  с нуля; `examples/train_tui.rs` — то же с **TUI-дашбордом прогресса** (ratatui,
  live-прогресс по 4 режимам + винрейт); либо `examples/train_loop.rs` —
  **бесконечное self-play поколениями с тёплым стартом и гейтингом** — не деградирует.
  **Гейтинг комбинированный** (всё на одних наборах бросков, усреднённо по `eval_seeds`):
  кандидат принимается, только если (1) **бьёт лучшего голова-в-голову** с перевесом
  ≥`margin` (`head_to_head` — прямой матч / команда-vs-команда / FFA кандидат-vs-best
  через `winrate_*_vs`; эта линейка не насыщается, в отличие от винрейта против
  фиксированной эвристики) **И** (2) **не проседает против эвристики** (`cs ≥ bs −
  margin`, страховка от непереходности). Усреднение по нескольким сидам (`strength_avg`/
  `head_to_head_avg`) режет выборочный шум ≈ в √`eval_seeds` раз. Штатная остановка по
  **Ctrl+C** (доигрывает текущий режим, повтор — сразу); сохранение **атомарное** (temp →
  `rename`); аргументы `<поколений> <партий-обучения> <партий-оценки> <сидов-оценки>
  <нейронов> <префикс-файлов> <alpha>` (зависит от dev-зависимости `ctrlc`, на WASM не
  влияет). Крупной сети с холодного старта полезен **больший α + больше партий** (напр.
  `… "" 20000 800 4 48 model_h48 0.15`).
  По умолчанию пишет `frontend/src/model*.bin` (после — `trunk build`). **Эксперимент с
  размером сети:** задать `<нейронов>` (напр. 48) И **свой** `<префикс-файлов>` (напр.
  `model_h48`), чтобы холодный старт создал сеть нужного размера в **отдельные** файлы,
  не затирая базовые (тёплый старт берёт размер из существующего файла). Сравнить два
  набора голова-в-голову по всем режимам — `examples/compare.rs`
  (`cargo run --release --example compare <prefixA> <prefixB> [партий] [сидов]`). **Self-play распараллелен**
  (`rayon`, фича `parallel` в `default`; GUI собирается с `default-features = false`,
  поэтому в WASM rayon не тянется): мини-батчи партий играются на всех ядрах,
  TD-апдейты — по порядку (детерминированно). С обогащёнными признаками
  (`blots`/`contact`) + h=48 + поколенным гейтингом MLP **уверенно бьёт** эвристику:
  2p ~70%, 3p ~50%, 4p ~35%, teams ~70% (раньше, на старых признаках, h=24/h=48
  застревали ~60% — узким местом были именно признаки). Это и есть рабочая модель GUI.
  (Демо-анимация в режиме разработчика по-прежнему на `Heuristic`.)
  Стороны-люди ходят кликами, компьютерные — сами; туровый цикл обобщён на множество
  людей (`humans` вместо одного `human` везде в кадрах/ходе). Очередь пропускает уже
  финишировавшие стороны (`next_unfinished_active` в движке).
  Доска — **векторная (SVG)** из двух слоёв: **статичный** (`static_board` — клетки
  периметра по `cell_kind`, слоты Дома в цвете стороны, дуги-параболы Луны со стрелкой
  и полями `1/3/6` — `moon_arc`, казематы Тюрьмы **внутри** доски, вплотную к клетке
  и сдвинутые к Дому — `prison_geoms`) и **слой фишек** — по одному keyed-кружку на
  фишку (`<For>`), позиция через `checker_xy` (клетка/дуга Луны/каземат, со сдвигом
  при стопке). В каземате у каждой активной стороны свой слот по длинной оси
  (`prison_slot`/`prison_slot_point`); одноцветные пленники накладываются в один
  кружок с **цифрой-счётчиком**. **Угол** — единственная (кроме Тюрьмы) клетка, где
  уживаются разные фишки; рисуются так же, как в Тюрьме (одноцветные — в один кружок
  со счётчиком), но разные цвета разнесены **наружу** от центра доски
  (`checker_corner`/`corner_sides`/`corner_slot_point`). **Поля Луны** стакаются так
  же (`moon_sides`/`moon_slot_point`); общий разнос наружу — `outward_slot`. Все
  цифры-счётчики одноцветных стопок (Тюрьма-каземат + углы + Луна + **клетка Тюрьмы**,
  где стоят освобождённые «зашёл-вышел» — `checker_on_prison`/`prison_cell_sides`,
  разнос как на углу) собирает `stack_counts`. Анимация хода — CSS `transition` на `cx/cy`
  keyed-кружков, причём ход (и человека, и ИИ) проигрывается **пошагово, по клеткам**:
  ход раскладывается в кадры (`Frame`: `commit_frames`/`apply_with_frames` — по кадру
  на каждую промежуточную клетку, плюс кадр «кости брошены»), которые `play`
  прокручивает с паузами (`gloo-timers` + `spawn_local`); на время анимации ввод и
  авто-бросок блокируются (`animating`), а перед первым ходом партии держится
  пауза-заставка (`kickoff`/`INTRO_MS`). Над доской — **статус-ведущий** (`herald`):
  каждый кадр несёт реплику (`note`: начало игры, бросок, съедание, выкуп, Луна,
  Тюрьма, Дом, финиш — `move_note`/`roll_note`; итог партии — `result_msg`). Клетка
  входа в Дом обводится цветом стороны только у **активных** домов (`home_side`);
  неиспользуемые дома не подсвечиваются. **Клетка входа в Дом (ворота) — обычная клетка
  периметра**, на которой можно остановиться (`Step`); сам заход в Дом — отдельный ход
  с неё, цель — слот Дома вглубь (`move_dest`/`dst_point` как обычно). **Лотков нет**: резерв,
  плен и кости вынесены **на доску, снаружи квадрата напротив Дома** каждой стороны
  (выносы `RESERVE_OUT`/`CAPTURED_OUT`/`DICE_OUT`, ряды через `outside_point`;
  `viewBox` расширен на `RESERVE_PAD`). Резерв и плен — обычные keyed-фишки (плен с
  красной обводкой), кликабельны через подсветку+хит в их зоне (`outside_anchor`).
  Бросок ходящей стороны — **3D-кубик** (`die3d`: 6 граней через CSS `preserve-3d`)
  кувыркается (анимация `die-roll`, флаг `rolling`) и встаёт нужной гранью (поворот
  через `--rx/--ry`), после чего показывается результат (`die_face`); **дубль** —
  золотая рамка (`.dice.double`). Кости — **HTML-оверлей** (`.board-dice` в
  `.board-wrap`), спозиционированный по координатам `viewBox` у Дома ходящей стороны.
  Бросок человека анимируется в начале его хода, броски ИИ — по ходу кадров
  (`commit_frames`, флаг `roll_anim`). Доска **масштабируется под окно**. Ход —
  **кликами** (фишка → клетка/зона) по одной кости с подсветкой; сопоставление —
  через `sheshbesh::checker_cell`. Вид — CSS-классы в `index.html`. Сборка — Trunk.
  **Настройки на лету** (`settings.rs`): **тема оформления** (`Theme`: Полночь/Рассвет/
  Лес — CSS-переменные на `:root[data-theme=…]`, применяются к `<html>` через
  `apply_theme`) выбирается иконкой-палитрой «🎨» с выпадающим списком (`theme_menu`) —
  и в **основном меню**, и в верхней панели **в партии**; **скорость анимации**
  (`Speed`: 5 положений — множитель пауз `factor` для `frame.hold` в `play` + CSS-
  переменная `--move-dur`) — **ползунок** (`<input type=range>`, `index`/`from_index`):
  на **главном экране** показан сразу (инлайн, `SpeedControl` в `.set-row`), а в партии
  — в **выпадающем меню по иконке «⏩»** (`SpeedMenu`, не прерывает игру). На
  максимальной скорости (`VeryFast`) бросок **без анимации** — кости появляются
  мгновенно; это касается и хода компьютера (`instant_roll` в `play`), и **броска
  игрока-человека** (паузы броска масштабируются `factor`, на `VeryFast` — мгновенно).
  **Звук** — иконка-тумблер
  «🔊/🔇» (`SoundKind`, синтез Web Audio без файлов). Звуки строятся из лёгкого
  синтез-набора в `settings.rs`: `voice` (осциллятор → **лоупас**-фильтр → огибающая —
  фильтр срезает резкие гармоники меандра/пилы) и `noise` (короткий буфер белого шума
  → полосовой фильтр — для «стука» костей и удара съедания). События: бросок/ввод/
  съедание/в-из Тюрьмы/Луна/выкуп/Дом/победа — по `frame.rolling` и реплике
  `note_sound`. **В режиме разработчика** есть панель прослушивания (`SoundKind::ALL` —
  по кнопке на каждый звук). Значения хранятся в `localStorage` (`load`/`save_*`).
  Сами элементы управления вынесены в **переиспользуемые компоненты** (`controls.rs`):
  `ThemeControl` (палитра «🎨» + меню), `SoundControl` (тумблер «🔊/🔇»), `SpeedControl`
  (ползунок) и `SpeedMenu` (иконка «⏩» с выпадающим `SpeedControl`); сигналы
  (`theme`/`theme_menu`/`sound`/`speed`/`speed_menu`) общие на `App`, поэтому выбор
  держится при переходах. Эти компоненты — в **основном меню**, **в партии** и **на
  экране обучения** (там же кнопка «📖 Правила»; на время Правил экран обучения скрыт,
  как и игровой). **Горячие клавиши** (`window_event_listener`, по `e.code()` —
  не зависит от раскладки): `Escape` — **послойно**: закрывает открытое всплывающее
  окно (лог/скорость/тема/Правила/«От автора»), а если ничего не открыто и идёт
  обучение — выходит из обучения в меню (так из открытых в обучении Правил первый
  Escape вернёт к обучению, второй — в меню); `s` — звук, `t` — следующая тема по
  кругу (`Theme::next`), `p` — пауза/продолжить **только в партии** (буквы игнорируются
  при фокусе в поле ввода). Зависимости: `web-sys` (Audio + DOM), `wasm-bindgen`.
- `src-tauri/` (`sheshbesh-tauri`) — Tauri v2-оболочка: нативное окно вокруг
  фронтенда (IPC не используется — логика во фронтенде). Запуск: `cargo tauri dev`
  (нужен `cargo install tauri-cli`); `tauri.conf.json` сам зовёт `trunk` для сборки
  фронтенда. Иконки в `src-tauri/icons/`.

## Команды

- Сборка: `cargo build` (релиз: `cargo build --release`)
- Запуск: `cargo run` — в настоящем терминале запускает интерактивную игру в
  ratatui-TUI (человек против эвристики); при выводе в пайп/лог переключается на
  ANSI-анимацию партии ИИ (выбор по `std::io::stdout().is_terminal()`).
- Управление в TUI: ход собирается по одной кости за шаг — `↑/↓` выбирают вариант
  (слева превью результата на доске), `Enter` — применить кость, `q`/`Esc` — выход.
- Окружение: `SHESHBESH_HUMAN` — стороны человека (`A`, или `A,C`; по умолчанию `A`);
  `SHESHBESH_DELAY_MS` — пауза после хода ИИ; `SHESHBESH_MAX_TURNS` — предел для ANSI;
  `SHESHBESH_SCALE` — ручной масштаб доски 1..4 (по умолчанию TUI подбирает сам под
  размер окна); `SHESHBESH_ASPECT` — соотношение высота/ширина символа терминала
  (по умолчанию определяется из размеров окна в пикселях, иначе 2).
- Все тесты: `cargo test` (движок + TUI)
- Один тест: `cargo test <имя_теста>` (`-- --exact` для точного совпадения)
- Движок без TUI / WASM: `cargo build --no-default-features [--target wasm32-unknown-unknown]`
- GUI (Leptos): из `frontend/` — `trunk serve` (dev-сервер) или `trunk build`;
  линт фронтенда — `cargo clippy -p sheshbesh-gui --target wasm32-unknown-unknown`
- Линтер: `cargo clippy --all --workspace --all-targets` — включён `clippy::pedantic`
  (через `[lints.clippy]` в `Cargo.toml`); шумные для этого кода правила (числовые
  приведения `as`, `must_use` на геттерах, `similar_names`) явно разрешены там же.
- Форматирование: `cargo fmt`

Edition 2024 требует Rust 1.85+. Зависимости: `ratatui` + `crossterm` (только для
TUI; движок/ИИ от них не зависят).

## Правила игры (самобытный вариант — авторитетная спецификация)

Это **не** стандартные нарды. Реализуйте именно эти правила.

> **«Выбросить» конкретное число всегда означает значение на одной кости.** Везде, где
> правило требует «выбросить» определённое число (**6** — чтобы ввести фишку, выкупить
> или ввести повторно; **4** — чтобы выйти из Тюрьмы; **1 / 3 / 6** на Луне), одна из
> двух костей должна *показать это значение* — сумма костей не засчитывается.

### Доска

- Квадрат. Каждая из 4 сторон содержит **19 клеток**, включая угловые на концах.
- **Углы общие** для двух смежных сторон, поэтому периметр = 4×19 − 4 = **72 клетки**.
  В абсолютной нумерации против часовой стрелки углы находятся на индексах 0, 18, 36, 54.
- **Посередине каждой стороны** находится **Дом (Home)**: 4 клетки, идущие *внутрь*
  квадрата. Каждый игрок должен загнать в Дом свои 4 фишки.
- Фишки обходят квадрат **против часовой стрелки** (полный круг), прежде чем войти в Дом.
- **Углы** — особые клетки, где фишки могут соседствовать (съедания там нет — см. ниже).

**Число игроков.** Играют **вдвоём или вчетвером**. Каждый игрок владеет одной стороной
и её Домом. При игре вдвоём используются **противоположные** стороны.

**Локальные индексы стороны (0..18; 0 и 18 — углы):** Луна — индекс **2**; две Тюрьмы —
индексы **4** и **14** (4-я клетка от каждого угла); вход в Дом / точка ввода фишки —
индекс **9** (середина); выход Луны — индекс **16** (обычная клетка, симметричная входу
относительно середины). На сторону приходится **одна Луна и две Тюрьмы**.

### Особые клетки (отсчёт от угла вдоль стороны)

- **Луна (Moon)** — **2-я клетка от угла**. Попав на неё, фишка «улетает» внутрь
  квадрата, на внутреннюю дорожку Луны. Луна — это «срез»: при удаче позволяет фишке
  быстро пройти сторону квадрата.
  - Внутренняя дорожка состоит из **3 круглых полей** с надписями **«1»**, **«3»**,
    **«6»**, проходимых по порядку — каждое продвигает только при соответствующем
    значении кости:
    1. Попав на Луну, фишка встаёт на поле **«1»**.
    2. На **«1»** выброс **1** → переход на поле **«3»**.
    3. На **«3»** выброс **3** → переход на поле **«6»**.
    4. На **«6»** выброс **6** → выход с Луны на **2-ю клетку от угла** (клетка стороны,
       **симметричная относительно Дома**). Если на клетке выхода стоит чужая фишка, она
       **съедается**.
- **Тюрьма (Prison)** — **4-я клетка от угла** (2-я клетка за Луной). Попадание сюда
  **изолирует** фишку. Единственный выход — выбросить **4**; фишка при этом **остаётся
  на той же клетке** (никуда не продвигается) и на выходе **не ест**. Отсюда тактика:
  стоять на клетке Тюрьмы и блокировать ею чужие фишки.
  - **Войти или мимо — определяется конечной клеткой движения.** Фишка попадает в Тюрьму
    (`EnterPrison`) только если клетка Тюрьмы — **конечная** клетка её хода. Если фишка
    лишь **перешагивает** клетку Тюрьмы по пути (значение кости перелетает её, или кости
    **объединены** в один мах — см. «Структура хода»), Тюрьма **не срабатывает** — это
    обычный `Step`. Поэтому отдельного «прохода насквозь» нет: чтобы пройти мимо, фишка
    просто не **останавливается** на Тюрьме. Можно также **зайти одной фишкой, а другой
    сходить** оставшейся костью — это легально (соблюдалось бы только правило максимума).
  - **Зашёл-вышел (кость «4»).** Зайдя в Тюрьму, костью **4** делается «зашёл-вышел»:
    фишка освобождается и **остаётся на клетке Тюрьмы** (`PrisonRelease`, свободна, на
    выходе не ест).
  - **Тюрьма симметрична Луне.** Если перешагнуть Тюрьму нечем (единственная фишка, а
    Тюрьма ровно на бросок впереди — дубль 5-5, где одиночная 5 встаёт ровно на неё),
    фишка **садится в Тюрьму и ждёт «4»** — ровно как фишка, влетевшая на Луну, ждёт
    нужных 1/3/6. Объединение костей (5+5=10 «в один мах») при этом **перешагивает**
    Тюрьму, и правило максимума выбирает перелёт; если же и объединённого пути нет —
    заход неизбежен. Раньше для этого был костыль «проход насквозь» — теперь не нужен.
  - Фишка, сидевшая в Тюрьме с прошлого (или этого) хода, выходит только по «4».

### Структура хода

- За ход игрок кидает **две кости**.
- Право первого хода: начинает тот, кто перед розыгрышем первым выбросил **6**.
- Все фишки стартуют в Доме и **вводятся в игру** выбросом **6** (по одной 6 на каждую
  вводимую фишку). **Нельзя ввести фишку, если на клетке ввода уже стоит своя фишка**
  (две свои на точке ввода не ставятся); **чужую** на клетке ввода вводимая фишка
  **съедает** (как обычное съедание).
- Два значения костей можно отнести к **двум разным фишкам** (по значению на каждую)
  **или** объединить на **одну фишку** — на усмотрение игрока. При объединении («в один
  мах», `Step` на сумму костей) **срабатывает только конечная клетка**: спец-клетки
  (Луна/Тюрьма), которые фишка лишь **перешагивает** по пути, не срабатывают (вот почему
  «проход насквозь» не нужен). Чтобы **зайти** на спец-клетку, её надо сделать **конечной**
  (одиночной костью или объединением). Промежуточные клетки всё равно должны быть
  свободны (правило «нельзя проходить через фишки»; занятая Тюрьма этим и блокирует).
  Объединять очки можно только при ходе по периметру; спец-состояниям (резерв/Луна/
  Тюрьма/плен/Дом) нужна ровно **одна** кость нужного значения.
- **Дубль** даёт право на **ещё один полный ход** (переброс костей и новый ход).
- **Правило максимального хода**: из допустимых вариантов игрок обязан выбрать тот,
  который использует **наибольшее число очков** (в идеале — все).
- **Нельзя проходить через фишки**: при ходе все промежуточные клетки периметра
  (не считая конечной) должны быть **свободны** от любых фишек — и своих, и чужих.
  Исключение — **углы**: через занятый угол проходить можно. Конечная клетка: чужую
  фишку **съедаешь**; на клетку, где уже стоит **своя** фишка, встать **нельзя** (двух
  своих на одной клетке не бывает) — **исключения только угол и Тюрьма** (вход/«зашёл-
  вышел»), где фишки соседствуют. **Союзники** (команды 2×2) — не «свои»: их фишки
  могут сосуществовать на одной клетке. **Дом — коридор вглубь:**
  клетки Дома проходятся по порядку (глубина 0 — у входа), и **перепрыгивать занятые
  клетки Дома нельзя** (ни при заходе, ни при движении внутри). Чтобы партия
  оставалась завершимой, фишка в Доме может **идти глубже** (`HomeAdvance`),
  освобождая мелкие клетки для входа других — поэтому занятая мелкая клетка не
  навсегда блокирует Дом. Перебор за пределы Дома **нелегален**; клетка входа (ворота)
  — обычная клетка, на ней можно остановиться, а заход в Дом — **отдельный ход** с неё
  (Дом начинается на шаг ЗА воротами). Но **второй круг возможен через Луну**: своя Луна стоит у самого
  Дома (прогресс ~65), а выход с неё симметричен относительно Дома и приходится на
  начало круга (малый прогресс), так что фишка отправляется на ещё один круг.

### Съедание («поедание»)

- Фишка съедает фишку противника, попав с ней на **одну клетку**.
- **Съедания нет** на: **углах**, в **Тюрьме** и на **Луне**.
- **Союзники не едят фишки друг друга** (командный режим 2×2): фишки союзных сторон
  могут сосуществовать на одной клетке. В движке отношение задаётся `GameState.teams`
  (`with_teams`) + `GameState::are_allied`; единственная точка съедания `enemies_on`
  исключает союзников, а ИИ (`opponents_of`) не считает их угрозами.
- **Луна полностью безопасна** (и вход, и внутренняя дорожка): на клетке-входе стоять
  нельзя — фишка сразу улетает внутрь, а внутри её съесть нельзя.
- Съеденная фишка **изымается из игры** и попадает в плен к сопернику (выкуп):
  - Чтобы **выкупить** её, нужно выбросить **6** — и тогда соперник обязан сделать
    дополнительный ход на **6 клеток**. Если полный ход на 6 клеток могут сделать
    несколько фишек соперника, он выбирает, какой. Если ни у одной фишки нет легального
    хода на 6 клеток, обязательный ход пропускается.
  - Затем нужна ещё одна **6** для **повторного ввода** этой фишки в игру.
  - Соперник **не имеет права** отказать в выкупе.

### Цель

Завести все фишки в свой **Дом**. Вражеские фишки в Дом проникнуть не могут.
Дом — 4 клетки; заход **ровно по броску, по одной фишке в клетку** (перебор за
пределы Дома не заводит фишку). **Клетка входа в Дом (ворота) — обычная клетка
периметра** (`OnTrack { progress == PERIMETER }`, т.е. 72): пройдя полный круг, фишка
**может остановиться на ней** обычным ходом (`Step`; на воротах можно стоять, как на
любой клетке — съесть чужую, не вставать второй своей). **Заход в Дом — отдельный ход**
уже с ворот (или «насквозь» через них): глубина 0 — на шаг ЗА воротами
(`target == PERIMETER + 1`), глубина 3 — `PERIMETER + 4`; дальше **перебор**, нелегально.
**Дом — коридор вглубь** (`MoveKind::EnterHome` — заход, `HomeAdvance` — движение
глубже): **перепрыгивать занятые клетки Дома нельзя** (все клетки от глубины 0 до
целевой свободны), но фишка в Доме может идти глубже, освобождая мелкие клетки, — так
Дом не «запирается» намертво. Заход в Дом **пересекает клетку входа** (ворота), только
если фишка идёт В ДОМ *из-за* ворот (`target > PERIMETER && progress < PERIMETER`) —
тогда занятые (хоть своей, хоть чужой фишкой) ворота **блокируют заход** (их нельзя
перепрыгнуть, они не угол; `path_clear`). Если же фишка УЖЕ стоит на воротах
(`progress == PERIMETER`), она их не пересекает и заходит свободно.

## Структура кода

- `src/board.rs` — геометрия: `Side`, `PerimeterIdx`, `cell_kind`, константы и
  локальные индексы; `Side::entry`/`progress_of` для перевода клетки ↔ прогресса.
- `src/dice.rs` — `Die`, `DiceRoll` (`has_value` — проверка значения на одной кости).
- `src/state.rs` — `Position`, `MoonField`, `Checker`, `GameState`.
- `src/moves.rs` — движок ходов: `legal_moves`, `legal_turns` (с правилом
  максимального хода), `apply`, `max_pips`. `Move` продвигает фишку на `pips` — значение
  **одной** кости или **сумму** объединённых (`advance_on_track` — ход по периметру «в
  один мах»: эффект только у конечной клетки, спец-клетки на пути перешагиваются;
  `resolve_special` — спец-состояния, ровно одна кость). `maximal_sequences` перебирает
  и одиночные ходы, и объединения (`motion_candidates`). Из-за объединений **максимальные
  ходы бывают разной длины** (`[Step{a+b}]` и `[Step{a},Step{b}]` оба максимальны) — код,
  собирающий ход по кликам, опирается на «нечем продолжить», а не на фиксированную длину.
- `src/turn.rs` — туровый цикл: `Game`. `play_turn` играет ход **по шагам**: после
  каждого выкупа СРАЗУ применяется обязательный ответ захватчика, и **только потом**
  игрок (агент) решает остаток костей — поэтому после выкупа состояние и остаток
  пересчитываются (`legal_turns_remaining`/`remaining_after`). Так решение о второй
  кости (напр. о втором выкупе) принимается уже после ответа, а не заранее на весь
  ход (тест `player_decides_after_forced_reply`). `commit_turn(roll, played, forced)` —
  применение **заранее выбранного** хода целиком (выкупы, дубль-переброс, передача
  очереди); используется туториалом GUI. `winner`; `start`/`determine_first` —
  розыгрыш права первого хода); `Agent` (выбор хода и вынужденного ответного хода)
  с `FirstChoice`; `DiceSource` с `ScriptedDice` (тесты) и `RandomDice` (встроенный
  SplitMix64, без внешних зависимостей).
- `src/render.rs` — компактный текстовый `render(&GameState) -> String` (токены
  `A2`) плюс **общая сетка глифов** `board_glyphs` (`BOARD_MARGIN` полей вокруг
  квадрата; первая фишка клетки — на клетке, лишние «дописываются» наружу,
  переполнение — `Overflow`). **Дом и Луна рисуются внутрь квадрата**: слоты Дома
  (`o`, от входа активных сторон) и поля Луны (`1/3/6`, от входа на Луну каждой
  стороны), занятые — маркером фишки. **Луна** (`Glyph::Moon`, свой цвет): вход
  `M`, поля `1/3/6` внутрь, выход `m`. **Тюрьма** (`Glyph::Prison` — клетки `J`;
  `Glyph::Captive` — пленённая фишка на красном фоне). Масштаб (`cell_size`):
  клетка = `cols × rows`, где `cols/rows ≈ аспект символа` (`cell_aspect` — из
  размера окна в пикселях через `crossterm::window_size`, иначе 2), чтобы доска была
  **физически квадратной**; `app` подбирает размер под ширину **и высоту** области,
  `tui` — компактный. `SHESHBESH_SCALE`/`SHESHBESH_ASPECT` переопределяют. На этой
  сетке строят доску и `tui`, и `app` (через `Glyph`/`margin_coord`).
- `src/tui.rs` — потоковая ANSI-анимация (`play_animated`): цветные кадры с
  очисткой экрана поверх `board_glyphs`; работает без TTY (удобно пайпить в лог).
- `src/app.rs` — TUI на `ratatui`/`crossterm`: `run_interactive` (игра человека) и
  `run` (авто-проигрывание ИИ). Человек собирает ход **по одной кости за шаг**
  (`human_pick_sequence` + `step_options`): варианты каждого шага берутся только из
  `legal_turns`, поэтому правило максимального хода (за обе кости) соблюдается
  автоматически; слева — список ходов, справа — текущая позиция с подсветкой
  выделенного хода (откуда — синим, куда — зелёным; `board_lines_hl` +
  `move_endpoints`). Доска рисуется с внешними полями (`MARGIN`): несколько фишек
  на одной клетке «дописываются» наружу за край (сверх — `+`). Вынужденный ответ при
  выкупе тоже выбирается интерактивно, если захватчик — человек. Решения человек/ИИ
  маршрутизируются вокруг `Game::commit_turn`. Чистая логика тестируется без TTY.
  Панель статуса (`panel_lines`) показывает резерв/Дом/плен **маркерами фишек**,
  равнозначные варианты хода схлопываются (`dedup_moves`).
- `src/ai.rs` — `Heuristic`: жадный агент (оценка на один ход вперёд). Ценит
  продвижение к Дому и съедание, штрафует плен/Тюрьму и **риск быть съеденным**
  (плюс слабее — свои угрозы чужим открытым фишкам). Оценка угроз (`shots_to`)
  считает реальные броски из 36 **с учётом блокировки** (свободный путь, проходимые
  углы, проходимая промежуточная остановка для комбинаций). `Heuristic` реализует
  `value::Value` (ручная оценка `evaluate`), а его `Agent` — это `best_turn`/
  `best_forced` над этой оценкой.
- `src/value.rs` — абстракция оценки для эвристики **и обучаемого ИИ**: trait
  `Value` (ценность позиции для стороны), выбор хода по ценности **афтерстейта**
  (`best_turn`/`best_forced`/`apply_sequence` — афтерстейт детерминирован, кости уже
  известны; случайность следующего броска `Value` усредняет неявно, приём
  TD-Gammon), и `ValueAgent<V>` — `Agent` поверх любой `Value`.
- `src/search.rs` — **2-плай expectiminimax** поверх любой `Value` (`best_turn_2ply`,
  агент `TwoPly<V>`). Дерево MAX (мы) → CHANCE (21 взвешенный бросок соперника) →
  MAX (соперник), лист — `V(позиция, мы)`. Свои ходы предварительно отсеиваются по
  1-плай (top-`DEFAULT_WIDTH`), чтобы не разворачивать соперника для слабых вариантов.
  Приближения: один ход соперника на бросок (дубль-переброс не раскрывается),
  вынужденные выкупы не моделируются, учитывается только **ближайший** соперник (для
  3/4 игроков поиск приближён). 2-плай поверх той же оценки играет не хуже 1-плай
  (тест `two_ply_beats_one_ply_heuristic`, `#[ignore]`, гонять с `--release`).
- `src/encode.rs` — кодирование позиции в вектор признаков `FEATURES` для обучаемой
  ценности с **канонизацией под ходящего**: стороны нумеруются относительно
  перспективы (`rel_owner = (owner−p) mod 4`, 0 — сам), клетки — относительно входа
  (`rel_cell = (abs−entry(p)) mod 72`). Блок на сторону: `pip` (нормированная гонка),
  `reserve`/`captured`/`prison`, `home[4]`, `moon[3]`, `track[72]` (контакт/блок —
  дорожка + Тюрьма), `mobile[6]`, `corner`, `blots` (доля своих одиночек на бьющихся
  клетках) и `contact` (доля блотов под прямым боём врага в 1..6 по ходу; путь/кость не
  проверяются — дешёвый сигнал, точное добирает `track`). `encode`/`encode_into`. `rotate_state(state, r)` поворачивает
  партию на `r` мест (аугментация ×4 и проверка инвариантности: `encode(s,A) ==
  encode(rotate(s,r), A+r)`). Отсутствующие стороны (2/3 игрока) — нулями.
- `src/eval.rs` — мерило силы агентов: `play_game_2p` + `match_winrate` (2 игрока,
  чередование мест) и обобщённые на N игроков `play_game` (диспетчер хода по стороне),
  `winrate_ffa` (модель на одном месте, остальные — `Heuristic`; честная доля `1/n`),
  `winrate_teams` (модель против эвристики в командах 2×2; честная доля 50%). **Нарды —
  игра с высокой дисперсией**, для надёжного сигнала нужно ~200+ партий (на ~40
  перевес тонет в шуме); эвристика берёт у `FirstChoice` ~65–70%.

- `src/learn.rs` — обучаемая ценность и TD(λ) self-play (подход TD-Gammon, чистый
  Rust). Trait `TdModel: Value` (умеет `td_update`); обобщённый тренер
  `train_with<M: TdModel>` (self-play текущими весами → траектория афтерстейтов →
  TD(λ) по подпоследовательности каждой стороны: γ=1, бутстрап к своему следующему
  афтерстейту, терминал 1/0). Затухание шага (`alpha_decay`) гасит расхождение TD
  при дрейфе self-play-политики (без него модель деградировала к ~3000 партий).
  Модели (обе импл `Value` + `TdModel`, `to_floats` для экспорта в GUI):
  - `LinearValue` = `sigmoid(b + w·φ)`. `train(&TdConfig)`. Устойчиво обыгрывает
    `Heuristic` **~58–61%** (800-партийный матч); ~13с на 10k партий (release).
  - `MlpValue` — однослойный перцептрон `sigmoid(b2 + w2·sigmoid(b1 + w1·φ))`
    (backprop через те же TD(λ)-следы; случайная инициализация `SplitMix`).
    `train_mlp(hidden, &cfg)`. Ловит нелинейности (контакт/угрозы). **Итог: после
    добавления тактических признаков (`blots`/`contact`) MLP h=48 заметно сильнее и
    линейной, и старого MLP** (2p ~70% против `Heuristic`); пока признаков не было,
    нелинейность было не на чем применить и MLP лишь сравнивался с линейной. Рабочая
    конфигурация GUI: **h=48, α≈0.15**, обучение `train_loop` (поколенный гейтинг).
    Тесты `trained_beats_first_choice` / `trained_mlp_beats_heuristic` (`#[ignore]`,
    гонять с `--release`).

**ИИ с обучением (RL), статус.** Подход TD-Gammon: обучение `Value` (ценности
афтерстейтов) в self-play через TD(λ), чистый Rust (тот же forward pass — и в движке,
и в wasm-GUI). Сделано: (0) `Value`/`ValueAgent` + эвал-харнес; (1) кодирование с
канонизацией под ходящего (4-кратная симметрия); (2) **линейная `Value` + TD(λ),
бьёт `Heuristic` ~58–61%**; (3) **MLP** реализован — но при текущем бюджете лишь
сравнивается с линейной (~57–60%), не превосходит (игра определяется гонкой; выигрыш
от нелинейности требует обучения масштаба TD-Gammon); (5) **3/4 игрока и команды
сделаны** — `TdConfig.active`/`teams`, обучение per-player (`winners_of`), модели
бьют эвристику: 3p FFA ~39% (честно 33%), 4p FFA ~28% (25%), команды ~57% (50%); (6)
**экспорт в GUI сделан** — компьютер ходит обученной моделью под текущий режим
(MLP-веса `frontend/src/model{,_3p,_4p,_4p_teams}.bin`, линейные —
`model_lin{,_3p,_4p,_4p_teams}.bin`; выбор `ai_model_for(algo, …)` по выбранному в
настройках алгоритму — **MLP/Linear/эвристика на каждого компьютера**, см. `algos`).
(7) **обогащены признаки** (`blots`/`contact`) — это сняло потолок: MLP h=48 теперь
уверенно бьёт эвристику (2p ~70%, 3p ~50%, 4p ~35%, teams ~70%) и стал рабочей
моделью GUI. Дальше (по желанию): ещё признаки и/или больший масштаб обучения.

**Модель положения.** Клетка на периметре кодируется как `OnTrack { progress }` —
число шагов (0..=72) от точки ввода **владельца**; абсолютная клетка и пройденный круг
выводятся из `progress`. `progress == 72` — фишка стоит на **клетке входа в Дом
(воротах)**, пройдя полный круг (это обычная клетка, на ней можно остановиться); сам
Дом — `Home { depth }` (глубина 0 — на шаг за воротами, заход отдельным ходом).

**Выкуп пленных** реализован: `Captured` + «6» → `Reserve` (`MoveKind::Ransom`,
участвует в правиле максимального хода), **сразу после чего** захватчик делает
обязательный ответный ход на «6» (если есть; иначе пропуск), и **только потом** игрок
решает остаток костей (см. по-шаговый `play_turn`). `TurnOutcome` содержит `applied` —
фактический порядок применения (ход игрока, за выкупом — ответ захватчика), по нему GUI
и анимирует. Повторный ввод фишки в игру — `Enter` по «6». **В GUI ход компьютера тоже
по-шаговый** (`play_computer` в `app.rs`): после выкупа компьютер выбирает остаток уже
после ответа; если захватчик — **человек**, ход приостанавливается (`forced_pick`) и
человек сам выбирает обязательный ответ, после чего компьютер продолжает остатком
костей. **Ход человека-выкупающего тоже по-шаговый** (`apply_part` в `app.rs` +
сигнал `remaining`): когда **человек** делает выкуп (`MoveKind::Ransom`), ответ
захватчика-компьютера применяется и анимируется **сразу** (`best_forced` под алгоритм
захватчика из `algos`), затем `turn_start` сдвигается на пост-форс-состояние,
`turns` пересчитываются `legal_turns_remaining` от остатка костей, а человек решает
остаток уже после ответа. Остаток костей хода ведётся в сигнале `remaining`
(`remaining_after` на каждый шаг) — по нему `finish` и считает «сгоревшую» кость.
Выкуп-последовательности **не сводятся в «обе кости сразу»** (`combo_targets`
исключает `Ransom`), чтобы ответ захватчика всегда вклинивался между костями. (Один
нюанс: `GAMELOG` для хода-выкупа человека пишет лишь пост-выкупный сегмент — debug-
мелочь.)

**Розыгрыш права первого хода** реализован (`determine_first`/`Game::start`):
активные стороны по очереди кидают кости, первая с «6» начинает.

Обязательный ответный ход при выкупе трактуется как **любой** легальный ход
захватчика на «6» (включая ввод фишки), а сам ход применяется после хода-выкупа.
Все правила из спецификации реализованы; представление — текстовое (без GUI).
